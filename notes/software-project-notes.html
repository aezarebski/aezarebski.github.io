Software Engineering

Documentation

Documentation, as described by divio, can be catagorised into four different types: tutorials, how-to guides, reference, and explanation. The goals and consequently structure of these types of documentation differ.

  • tutorials are like teaching a child to cook,
  • how-to guides are like a recipe in a cookery book,
  • reference material is like the wikipedia article on cooking,
  • explanation is like a commentary culinary social history.

Minimal generic structure (WIP)

.
├── ChangeLog.md (optional)
├── data
|   ├── processed
|   └── raw
├── LICENSE (essential)
├── out
|   └── plots
├── README.md (essential)
├── requirements.txt
├── src
└── test (optional)
  • Changelog.md seems like overkill but is a nice way to keep track of what has actually been done.
  • data is a directory for the raw data that is needed by the project
    • The raw subdirectory is for the original data set and processed is for tidy data
  • LICENSE seems like overkill but could be useful in the long run… advice from github ESSENTIAL
  • out is a place to write output to
    • The plots subdirectory is for plot output as opposed to other results
  • README.md ESSENTIAL.
  • requirements.txt or some way of recording the dependencies of the project: stack, packrat, virtualenv, …
  • src so the project can actually do something.
  • test so you can be confident that it does that thing properly.

Cockburn's use case template

Lifted from The Pragmatic Programmer

  • CHARACTERISTIC INFORMATION
    • Goal in context
    • Scope
    • Level
    • Preconditions
    • Success end condition
    • Failed end condition
    • Primary actor
    • Trigger
  • MAIN SUCCESS SCENARIO
  • EXTENSIONS
  • VARIATIONS
  • RELATED INFORMATION
    • Priority
    • Performance target
    • Frequency
    • Superordinate use cases
    • Subordinate use cases
    • Channel to primary actor
    • Secondary actors
    • Channel to secondary actors
  • SCHEDULE
  • OPEN ISSUES
